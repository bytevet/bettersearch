About BetterSearch

BetterSearch is a Chrome extension that replaces the browser's basic Cmd+F with a full-text search experience. It indexes the current page, ranks results by relevance using TF-IDF scoring, and lets users click through to highlighted matches.

Architecture Overview

The extension runs across four execution contexts defined by Manifest V3. The service worker handles command registration and side panel lifecycle. The content script performs DOM extraction, text indexing, and match highlighting directly on the page. The side panel provides a React-based search UI with components built on shadcn/ui and Tailwind CSS. Finally, the shared module contains type definitions and messaging helpers used by all other contexts.

How Indexing Works

When the side panel opens, it sends an INDEX_PAGE message to the content script. The content script traverses the DOM using depth-first search, extracting visible text segments from the page. Each segment records its text content, an XPath for locating the element, and a weak reference to the DOM node.

The extracted segments are fed into MiniSearch, a lightweight full-text search library that supports fuzzy matching, prefix search, and TF-IDF relevance scoring. Indexing happens in batches of 50 segments with progress updates sent back to the side panel via port-based messaging.

Once built, the index is serialized and cached in IndexedDB. On subsequent visits to the same page, the content hash is compared to detect changes. If the page content hasn't changed, the cached index is loaded instantly without re-traversing the DOM.

Search and Highlighting

When the user types a query, the side panel debounces the input by 150 milliseconds and sends a SEARCH message to the content script. MiniSearch returns results ranked by relevance score. Each result includes a snippet with HTML-escaped text and <mark> tags wrapping matched terms.

Clicking a search result sends a HIGHLIGHT message. The content script locates the target element via its XPath or weak reference, then uses a TreeWalker to find matching text nodes. Matched text is wrapped in <mark> elements with a yellow background. The first match receives an active class with a pulse animation and the page scrolls smoothly to bring it into view.

TypeScript and the Build System

The entire codebase is written in TypeScript with strict mode enabled. The build system uses Vite with three separate configurations: one for the side panel React app, one for the content script as an IIFE bundle, and one for the service worker as another IIFE bundle. A build orchestration script runs all three builds sequentially and copies the manifest and icon assets into the dist directory.

Testing Strategy

Unit tests use Vitest with jsdom for DOM simulation. The extractor tests verify correct text extraction from various DOM structures, including proper handling of hidden elements, script and style tags, and nested block elements. The indexer tests cover search accuracy, fuzzy matching, prefix search, relevance ranking, snippet generation with HTML escaping, and serialization roundtrips. Cache tests verify IndexedDB read and write operations, content hash validation, and URL normalization.

Component tests use React Testing Library to verify the SearchBar input behavior, debouncing, result count display, and the ResultsList click handling. End-to-end tests use Playwright with a persistent browser context that loads the extension via the --load-extension flag.

Security and Privacy

BetterSearch makes no network requests. All text processing happens locally within the page context. The extension requires only two permissions: activeTab for accessing the current tab's content, and sidePanel for the search UI. A strict Content Security Policy restricts script sources to the extension itself. Cached data in IndexedDB contains only text the user already sees on the page, and entries are automatically cleaned after seven days.

SPA Support

For single-page applications that dynamically update their content, BetterSearch installs a MutationObserver on document.body. When significant DOM changes are detected, a debounced timer (2 seconds) triggers the index to be cleared. The next search will rebuild the index from the current page state, ensuring results stay accurate even on highly dynamic pages.
